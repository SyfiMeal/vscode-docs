## Agents

### Visual Studio Code Agents (Insiders)

> **Note**: Visual Studio Code Agents is currently in preview and only available with VS Code Insiders.

[Visual Studio Code Agents](https://aka.ms/VSCode/Agents/docs) is a companion that ships with VS Code Insiders. It provides a focused, agent-native environment where you can run parallel sessions across repos and iterate on multi-step coding tasks. We first introduced VS Code Agents in [1.115](https://code.visualstudio.com/updates/v1_115#_visual-studio-code-agents-preview) and continue to refine it based on user feedback.

Updates in this release include:

* **Redesigned new session repo picker**: Starting a new session provides a streamlined repo picker that makes it easier to find and select the repo you want to work in.

  ![Screenshot of the redesigned repo picker in VS Code Agents.](images/1_119/Agents-new-dropdown-2.png)

* **Sub-session improvements**: Creating and managing sub-sessions continues to improve, with fixes in areas such as sub-session tabs and lifecycle handling.

* **Web and mobile polish**: We continue to iterate on the https://insiders.vscode.dev/agents web client introduced in [1.118](https://code.visualstudio.com/updates/v1_118#_web-client), with refinements so the browser experience stays in step with the desktop. This includes improvements to the mobile experience, so that you can create and manage sessions and their changes from the browser of your mobile device.

* **Environment management and continuity**: We continue investing in the connections between VS Code and Agents and evolving how the environment is managed - this will continue taking shape in future releases.

*  **Progress improvements**: Progress should now be much more obvious and apparent as the agent works, with rotating progress messages and light animations around the input box.

*  **Developer joy**: We're iterating on UX opportunities to spark developer joy, including a fun easter egg on the new session page. Enable `setting(sessions.developerJoy.enabled)` to see if you can spot it!

Your feedback helps us shape the Agents experience - please continue sharing it with us by [filing issues on GitHub](https://github.com/microsoft/vscode/issues). You can also explore [existing issues](https://github.com/microsoft/vscode/issues?q=state%3Aopen%20label%3A%22agents-app%22) to see what others have reported and provide your feedback on specific topics.

## Chat

### Live model and multiplier details for Copilot CLI and Claude agent

**Setting**: `setting(github.copilot.chat.agent.modelDetails.enabled)`,

[Copilot CLI](https://code.visualstudio.com/docs/copilot/agents/copilot-cli) and Claude agent responses in the Chat view show the last used model and the model multiplier on each response, so you can see at a glance which model handled each turn and how it counts against your usage. The badge appears live as each response completes, without needing to reload the window, and updates when you switch models mid-session.

When using **Auto** model selection in Copilot CLI, the badge displays the actual resolved model (for example, `Claude Sonnet 4.6`) rather than `auto`. The resolved model is also preserved when the session is rebuilt from history.

The behavior is enabled by default. To turn off the badge, disable the `setting(github.copilot.chat.agent.modelDetails.enabled)` setting and reload the window.

### Lightweight background agent to manage todo lists (Experimental)

**Setting**: `setting(github.copilot.chat.agent.backgroundTodoAgent.enabled)`

Reduce token usage during multi-step agent tasks by offloading todo list management to a lightweight background agent. Normally, keeping the todo list in sync with actual progress relies on the main model deciding to update it using tool calls. When this setting is enabled, a lightweight background agent with a smaller model monitors main agent activity and updates the todo list to reflect completed and in-progress work when necessary. The main agent will not have the todo tool available, thus saving token cost for the conversations.

> [!Note]
> If the todo tool is manually added to the chat request (for example with `#todo`), or a custom agent specificies it in its tool list, the background agent will be overidden and not run.

This feature is disabled by default. To try it out, enable the `setting(github.copilot.chat.agent.backgroundTodoAgent.enabled)` setting.

### Easier sharing of browser tabs with agents

For agents to see and interact with the [integrated browser](https://code.visualstudio.com/docs/debugtest/integrated-browser), browser pages must be explicitly shared. This helps keep sensitive data private, but it can be confusing.

In this release, we've added some new ways to share browsers with an agent:

#### Attach browsers as context

Browser tabs can now be explicitly attached to the chat via typical entry points such as suggested context, context picker, and drag-and-drop.

![An implicit context item in the chat, allowing the user to add the open browser to the chat context](images/1_119/browser-implicit-context.png)

When a browser tab is attached, it will enter a sharing state. The ![Sharing with Agent](images/1_119/browser-share-button.png) button can be toggled in the browser to stop sharing the page.

#### Agents can request to share a page

Agents can now see how many browser tabs you have open but have not shared, and can request to share an open tab.

Additionally, if an agent tries to open a new tab on the same domain as an existing, unshared tab, a prompt will appear to share the tab. This aims to encourage tab reuse and reduce clutter.

![A chat session showing the user prompting to "Test the app in the browser" and a resulting prompt to the user to "Share an existing tab?"](images/1_119/browser-sharing-prompt.png)

## Trust and Security

### Sandbox: Allow network access

Sandboxing now supports a new `allowNetwork` mode for `chat.agent.sandbox.enabled`:

```json
"chat.agent.sandbox.enabled": "allowNetwork"
```

This keeps agent tool sandboxing enabled, including file system restrictions, but disables sandbox-based network domain blocking. In this mode, all terminal commands remain sandbox-wrapped and no restriction to access any network domain. `chat.agent.allowedNetworkDomains` or `chat.agent.deniedNetworkDomains` do not have any effect on the network access.

Use this when you want the protection of the sandbox while allowing agent tools to access the network freely.

## Engineering

### Finished migrating webview's to use CSS anchor positioning

VS Code's webviews now use [anchor positioning](https://developer.mozilla.org/en-US/docs/Web/CSS/Guides/Anchor_positioning) to position themselves visually in the workbench. This improves performance and makes re-layouts more responsive, especially if there are many active webviews. It also let us fix some tricky, long-standing bugs, such as webviews going out of position on web when the workbench was moved.

Here's a typical re-layout call for a single webview before the switch to anchor based positioning:

![Perf chat of layout before anchor based positioning](images/1_119/anchor-perf-before.png)

Positioning the webview here was done using JS which called `getBoundingClientRect`. This call ends up being relatively slow because it triggers browser style recalculations and re-layouts.

By moving to anchor based positioning, now the browser handles positioning the webview for us based on the css.

![alt text](images/1_119/anchor-perf-after.png)


### All typechecking now uses TypeScript 7 for faster development iteration

Last iteration [we moved VS Code's main watch task to use TypeScript 7](v1_118.md#faster-development-builds-with-typescript-7). This iteration we finished the migration to use TypeScript 7 for all of our built-in extensions and core code.

Migrating the Copilot extension to use TypeScript 7 cut the typechecking time from 22 seconds to 4 seconds. These dramatic speedups allow both developers and agents to iterate more quickly in the VS Code codebase.
